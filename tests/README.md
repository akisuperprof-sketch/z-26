
# Project Z-26 自動回帰テスト基盤 v1

このディレクトリは、Project Z-26 の「正しい反応・正しい表示・正しい診断結果（SSOT）」を継続的に担保するためのテスト一式を格納しています。

## 1. テストの構成

### A) ゴールデン回帰テスト (Vitest)
`tests/regression/` 配下にあります。 `analyzeCore` の出力を `result_v2.output_payload` 形式でスナップショット比較します。
- **目的**: 推論ロジックや出力構造が不意に変更されていないことを保証します。
- **対象**: `result_v2.output_payload` (Single Source of Truth)

### B) E2E UIテスト (Playwright)
`tests/e2e/` 配下にあります。ブラウザ上で実際のユーザー操作をエミュレートし、画面遷移や権限ガードを検証します。
- **目的**: 診断フローの完走、医療断定表現の禁止、管理者専用画面の保護などを担保します。

## 2. 実行方法

### 全テスト実行
```bash
npm run test:all
```

### 回帰テストのみ（SSOTスナップショット）
```bash
npm run test:regression
```

### E2Eテストのみ
```bash
npm run test:e2e
```

## 3. テストケース一覧 (SSOT Golden)

| ケースID | 名称 | 入力内容の意図 | 期待される検証点 |
| :--- | :--- | :--- | :--- |
| **Case_01** | Yin Deficiency (Strong) | QUICK_LV4相当。明らかな陰虚の舌と設問回答。 | `top1_id` が陰虚系であること、`level` が高いこと。 |
| **Case_02** | Qi Stagnation | 舌辺の紅、イライラ・張りの回答。 | `top1_id` が `P_LIVER_QI_STAG` または関連証であること。 |
| **Case_03** | Neutral / Balanced | 極端な症状がない入力。 | `isNeutral` が `true`、`level` が低いこと。 |

## 4. スナップショットの更新ルール
ロジックや仕様の意図的な変更によりテストが落ちた場合のみ、以下のコマンドでスナップショットを更新します。
**注意: SSOTの構造変更は Phase1 では原則禁止です。更新前に必ず影響範囲を確認してください。**
```bash
npx vitest -u tests/regression/
```

## 5. 回帰テストで担保できること / できないこと

### 担保できること
- 同一入力に対して常に同一の判定結果（SSOT）が返ること
- UIの主要な遷移が壊れていないこと
- 本番環境でデバッグ機能が誤って有効にならないこと
- 明らかな医療断定用語の混入（簡易チェック）

### 担保できないこと
- 画像解析（Gemini）の精度の揺らぎ（Geminiの出力自体をモック化してテストするため）
- デザインの完全なドットレベルの維持
- すべてのパターンの網羅（現在は代表3ケース）
