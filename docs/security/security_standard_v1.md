# AG開発セキュリティ標準 v1.0 

## 1. 環境変数管理標準化
- **ローカル環境**: 全て `.env.local` のみで管理する。
- **バージョン管理**: `.env*` (例: `.env`, `.env.local`, `.env.test`) はすべて `.gitignore` の対象とし、Gitへのコミットを禁止する。
- **本番環境**: Vercel Environment Variables を使用して設定する。
- **コード直書き禁止**: APIキー、トークン、パスワードなどの機密情報をソースコード内にハードコーディングすることを固く禁ずる。

## 2. 機密検知自動化 (gitleaks)
- **初期導入スキャン**: 最初のみ `gitleaks detect --redact`（過去履歴含む全体スキャン）を一度実行し、結果0件（クリーン状態）を確認してから運用を開始する。
- **pre-commit hook (ローカル砦)**: `husky` を導入し、コミット時に自動的に `gitleaks protect --staged`（ステージングされた差分のみ）を実行する。
- **GitHub Actions (最終砦)**: GitHub Actions で push/PR 時に `gitleaks` ジョブを回すワークフローを設定し、ローカルでの hook バイパス（--no-verify等）や hook 無効環境でも確実にサーバー側で止める多段防御を構成する。
- **例外管理**: どうしても検知されてしまうが安全なダミー文字列等については `.gitleaksignore` にて個別除外ルールを設ける。

## 3. テストファイル管理標準
- **/tmp ディレクトリ運用**: デバッグ目的で出力される中間ファイルや一時的なテストファイルは、すべて `/tmp` ディレクトリ (例: `/tmp/z26-debug/`) に出力する。
- **出力ファイル管理ルール**: プロジェクトディレクトリ (ワークスペース) 内への出力は禁止。検証後に `/tmp` 内のファイルは自動/手動でクリーンアップする。

## 4. APIキー運用ルール
- **ローテーション**: APIキー (例: `INTERNAL_API_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `GEMINI_API_KEY`) は 90日ごとに定期的にローテーションする。
- **記録管理**: APIの最終更新日、次回更新予定日を `docs/security/key_management.md` で管理する。具体的なキー値は記載せず、メタデータのみを記録する。

## 5. 緊急時対応フロー (漏洩発生時)
1. **キーの即時無効化/再発行**: 対象プロバイダ (Vercel, Supabase, Google等) のコンソールから直ちに無効化し、新しいキーを発行する。（何よりも優先）
2. **影響範囲の特定**: どのファイル/ログから、どのキーが漏洩したかを特定する。
3. **本番環境の更新**: Vercel Environment Variables を新しいキーに更新し、再デプロイを行う。
4. **履歴削除 (BFG/filter-branch)**: `git filter-branch` または BFG を使用して、コミット履歴から該当情報を含むファイルを完全削除する。
5. **強制push基準**: 履歴削除後、チーム全員に周知した上でローカルを削除後の最新状態にし、 `git push --force --all origin` を実行する。
6. **事後報告**: 漏洩の原因究明と再発防止策を策定し、ドキュメントに記録する。
