# Z-26 バイブコーディング安全チェックリスト v1

本チェックリストは、AI(LLM)支援による高速開発（バイブコーディング）において、意図しないセキュリティインシデント（鍵漏洩、機密情報露出など）を未然に防ぐためのゲートウェイです。手動確認への依存を減らし、CIでの自動スクリプト(`scripts/security-scan.sh`)と併用して安全なリリースサイクルを回します。

## 1. 共通セキュリティ13項目（Z-26向け最適化）

| No | チェック項目 | Z-26での対応方針・静的検出基準 |
|:---|:---|:---|
| 1 | **ハードコードされた鍵の排除** | AIza(Gemini), supabaseキーなどがソースコード中に存在しないこと。`.env`経由のみを許可。 |
| 2 | **不要なログの削除** | `console.log` が本番(UIやAPI)で出力されないこと。(`tests/`, `scripts/`, `utils/debugConfig.ts` 等は除外) |
| 3 | **CORSの適切な制限** | `Access-Control-Allow-Origin: *` を使用しない。Vercel/Next.jsの標準または指定オリジンのみに制限する。 |
| 4 | **スタックトレースの隠蔽** | `res.status(500).json({ error: err })` 等、内部エラーの生オブジェクトをクライアントに返却しない。 |
| 5 | **オープンリダイレクト対策** | `redirect=` 等のクエリパラメータをそのままリダイレクトURLに渡さない。ホワイトリスト照合を必須とする。 |
| 6 | **環境変数の漏洩防止** | `.env`ファイルが`.gitignore`に含まれており、誤ってコミットされていないこと。 |
| 7 | **SQL / NoSQL インジェクション** | Supabaseクライアント経由のORMを正しく想定し、生クエリによる文字列結合を行わない。 |
| 8 | **XSS対策** | ユーザー入力（ヒアリング情報など）を直接 `dangerouslySetInnerHTML` 等でレンダリングしない。 |
| 9 | **CSRF対策 / セッション保護** | VercelとSupabaseの標準セッション管理に準拠する。過度な自前実装を行わない。 |
| 10 | **RLS (Row Level Security)** | データベース(Supabase)上で、匿名や別ユーザーによる不当なデータ読み書きを防ぐアクセス制御が掛かっていること。 |
| 11 | **データバリデーション** | API側で受け取るパラメータ（画像サイズ、型、文字数）に対する境界チェックを行う。 |
| 12 | **依存関係の脆弱性** | 古すぎるパッケージを積まない。必要に応じて `npm audit` を確認する。 |
| 13 | **権限エスカレーション防止** | AdminやPro昇格ロジックにおいて、クライアント側のフラグ（例: localStorage）だけを盲信しない。 |

## 2. Z-26 固有の保護要件

- **舌画像 (Tongue Image)**
  - 現在は外部保存せずローカル(Canvas)処理、もしくはセキュアなProxy APIを通じてAIへ送信後に即時破棄する。
- **年齢性別 (Personal Data)**
  - ヒアリングで取得した年齢や性別は分析目的のみに使用し、個人を特定可能な情報（氏名、連絡先）と紐づけて保存しない。
- **研究同意 (Consent)**
  - アプリ開始時に明示的に「同意チェック」を必須とし、これがない場合には解析APIや保存ロジックを走らせない。同意情報の有無（フラグ）をDBに記録する。

## 3. デプロイ前・運用ゲート手順

このチェックは人間の意志だけでなく、GitHub Actions (CI) により強制的に実行されます。

1. **開発者 (Agent / Dev)**
   - ローカルで `npm run test:all` と `bash scripts/security-scan.sh` が通ることを確認し、PRまたは `main` へ push する。
   - 変更が「不変領域（coreEngine、output_payloadなど）」に触れていないことを自己確認する。
2. **CI (GitHub Actions)**
   - Push時に `regression.yml` がトリガーされる。
   - インストール後に `npm run test:all` (Unit, E2E, Regression) を実行。
   - 続けて `bash scripts/security-scan.sh` による静的チェックを実行。
   - 1つでもFailすればデプロイプロセス（Vercel等の本番デプロイ）をブロックする。
3. **レビュー担当 (Owner)**
   - CIがオールグリーン（PASS）であれば、目視にてUI崩れやデザイン方針の逸脱がないか短時間のスモークテストを行う。
   - 問題なければマージ・リリースを許可する。
